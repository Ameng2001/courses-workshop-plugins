目标是把“教研工坊”重新定义成：

**Astra Studio 上可安装的课程设计行业插件集合。**

也就是说，本篇不再把教研工坊描述成一个独立平台，而是明确它如何：

- 注册自己的 task type
- 定义自己的 wizard
- 在中间区输出课程设计 block
- 在右侧 rail 中映射课程设计状态

---

## 2. 新定位

### 2.1 教研工坊是什么

教研工坊不再是平台壳，而是 Astra Studio 平台上的一个：

**课程设计行业插件集合**

它的职责不是提供：

- project shell
- task runtime
- global navigation
- customize / plugin management

而是提供：

- 课程设计相关 starter cards
- 课程设计相关 task types
- 课程设计相关 wizard
- 课程设计相关 render blocks
- 课程设计相关 context rail adapters
- 课程设计相关 outputs

### 2.2 教研工坊不是什麽

它不是：

- 一个单独的 AI 工作台产品
- 一个独立于 Astra Studio 运行的平台
- 一个重做了 project / task / rail 的二级壳子

---

## 3. 用户目标与核心价值

在平台层，用户是在做 task。
在教研工坊里，用户真正想完成的是：

### Persona A：课研主任

目标：

- 把一个主题想法变成完整课程结构
- 控制月矩阵、周节奏和活动类型平衡
- 在关键节点确认方向

### Persona B：一线教师

目标：

- 基于已有项目，补一节活动稿
- 调整已有 week 的内容
- 对齐客户模板并可执行

### Persona C：交付负责人

目标：

- 检查交付物是否齐全
- 准备 export bundle
- 对齐客户格式与模板

---

## 4. 行业插件集合的核心对象

平台层有：

- Project
- Task
- TaskType
- ContextSource
- Output

教研工坊在此基础上增加自己的领域对象。

### 4.1 Theme Package

主题结构包，包括：

- theme analysis
- theme narrative
- theme network

### 4.2 Month Matrix

月度活动矩阵，包括：

- 周次
- 子主题
- 教学活动
- 区域活动
- 户外游戏
- 生活渗透
- 家园互动

### 4.3 Week Arrangement

一周活动编排，包括：

- 线性活动序列
- 节奏
- 缺项
- 材料与教师备忘

### 4.4 Activity Draft

活动稿，包括：

- teaching activity
- region activity
- outdoor game
- life routine
- home-school

### 4.5 Export Bundle

课程交付包，包括：

- release bundle
- export manifest
- target-specific output prep

## 5. 教研工坊的 Starter Cards

行业插件集合安装后，平台空态和 starter selected 状态应增加教研 starter。

建议第一批只暴露高频的 4 张：

1. `创建一个主题课程项目`
2. `从月矩阵开始排结构`
3. `补一节活动稿`
4. `准备课程导出包`

### 5.1 创建一个主题课程项目

目标：

- 从一个主题想法起步
- 进入主题 framing task

### 5.2 从月矩阵开始排结构

目标：

- 基于已有主题方向，快速组织月度课程结构

### 5.3 补一节活动稿

目标：

- 针对已有 week / activity slot 快速生成活动稿

### 5.4 准备课程导出包

目标：

- 从已有成果快速组织交付 bundle

---

## 6. 教研工坊的 Task Types

教研工坊不应把整条课程链路压成一个超级 task，而应拆成几个稳定 task type。

### 6.1 `thematic-curriculum-framing`

目标：

- 把主题想法变成主题结构包

输出：

- theme analysis
- theme narrative
- theme network

### 6.2 `month-matrix-design`

目标：

- 设计月度活动矩阵

输出：

- month matrix
- 周次递进
- 覆盖与缺项提示

### 6.3 `week-arrangement-design`

目标：

- 把某一周排成可执行的节奏结构

输出：

- week arrangement
- day rhythm
- missing items
- materials / teacher notes

### 6.4 `activity-draft`

目标：

- 生成或修改一份具体活动稿

子类型可包括：

- teaching
- region
- outdoor
- life-routine
- home-school

### 6.5 `export-bundle-prep`

目标：

- 检查并准备课程交付包

输出：

- release bundle
- export bundle
- manifest

## 7. 教研工坊的 Wizard 设计

平台层提供 wizard host，教研工坊只定义各 task 的 wizard 内容。

## 7.1 `thematic-curriculum-framing` wizard

建议采用 hybrid wizard：

- 主题名称
- 年龄段 / 班级
- 课程类型
- 是否对齐客户模板
- 是否导入参考资料
- 默认 pipeline

wizard 完成后生成一段启动草稿，进入正式任务。

## 7.2 `month-matrix-design` wizard

字段建议：

- project
- 主题 / 子主题范围
- 是否已有 theme package
- 活动类型偏好
- 周数

## 7.3 `week-arrangement-design` wizard

字段建议：

- week
- 当前周重点
- 是否已有 month matrix
- 是否要按客户模板输出

## 7.4 `activity-draft` wizard

字段建议：

- activity type
- 归属 week
- 归属子主题
- 是否有客户模板
- 是否有参考样例

## 7.5 `export-bundle-prep` wizard

字段建议：

- target
- 是否准备客户交付版
- 是否需要 bundle summary

## 8. 中间区 Rich Blocks

教研工坊最重要的不是普通消息，而是课程结构块。

### 8.1 `theme-framing-block`

展示：

- theme analysis
- theme narrative
- theme network
- 四周递进摘要

### 8.2 `month-matrix-block`

展示：

- 月矩阵表
- 周次分布
- 活动类型覆盖

### 8.3 `week-rhythm-block`

展示：

- Mon-Fri 节奏
- 缺项
- 锚点活动

### 8.4 `activity-section-block`

展示：

- 目标
- 准备
- 重点难点
- 活动过程
- 教师观察与支持

### 8.5 `export-bundle-block`

展示：

- bundle tree
- manifest 摘要
- target 预览

### 8.6 `confirm-block`

教研任务中很多结果需要用户确认方向，但它不应是传统审批页，而应以 confirm block 出现在任务流中。

例如：

- 接受这版
- 继续细化
- 换一个方向

## 9. 右侧 Rail 映射

教研工坊应提供自己的 rail adapter，但仍然遵循平台统一骨架。

### 9.1 Progress

建议映射为：

- Theme Framing
- Month Matrix
- Week Arrangement
- Activities
- Export

注意：

- `Activities` 常是 1:N 展开
- rail 应支持显示当前阶段内部完成度

例如：

- `2 / 5 activities drafted`

### 9.2 Working folder

建议映射为：

- 当前 project
- 当前 pipeline
- 当前对象
- `.workshop/projects/...`

### 9.3 Context

建议展示：

- KB 样例
- client template
- current month plan
- current week plan
- referenced activity drafts

### 9.4 Outputs

建议展示：

- theme package
- month matrix
- week arrangement
- activity draft
- export bundle

---

## 10. 教研工坊的 OutputKinds

教研工坊行业插件集合应至少注册以下 output kinds：

- `theme-package`
- `month-matrix`
- `week-arrangement`
- `activity-draft`
- `course-release-bundle`
- `course-export-bundle`

每种 output 应声明：

- 是否 previewable
- 是否 downloadable
- 是否 project visible
- 是否进入 export bundle

## 11. 与现有 `course-workshop` 的关系

这一层不是取代当前 `course-workshop`，而是重新解释它在平台中的位置。

### 11.1 现在的 `course-workshop` 更接近什么

它已经具备：

- runtime model
- project / plan / export semantics
- domain-specific tasks
- HIL / review / promote logic

所以它不应再被理解为前端平台本体，而应被理解成：

**教研工坊行业插件集合的业务内核**

### 11.2 前端平台如何消费它

Astra Studio 平台通过：

- task types
- wizard
- render blocks
- rail adapters

把 `course-workshop` 的业务内核翻译成平台工作体验。

---

## 12. 行业插件集合与平台的边界

### 平台负责

- shell
- projects
- tasks
- execution surface
- task context rail
- customize / plugin management

### 教研工坊负责

- 课程 starter
- 课程 task types
- 课程 wizard
- 课程 rich blocks
- 课程 rail 映射
- 课程 outputs

---

## 13. 后续实现顺序建议

如果按新的平台层级推进，建议顺序是：

1. 先实现 Astra Studio 平台壳
2. 实现平台级 task runtime 与 rail
3. 再把 `course-workshop-plugins` 接成平台中的行业插件集合
4. 最后再做教研工坊的高保真任务流

而不应该继续把教研工坊的老页面原型当成平台壳去扩。

---

## 14. 结论

教研工坊在新架构里应被重新定义为：

**Astra Studio 上可安装的课程设计行业插件集合。**

它的角色是：

- 提供课程任务
- 提供课程 wizard
- 提供课程中间渲染块
- 提供课程右侧状态语义

而不是：

- 重新定义一套 project / task / shell / rail 平台。
